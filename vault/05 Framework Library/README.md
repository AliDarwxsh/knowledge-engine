# 05 Framework Library

> Frameworks are lenses. Pick the right one before you analyze.

## Purpose

This folder holds the operator's **mental models and thinking tools** — patterns of analysis that have proven useful across multiple domains. A framework is a way of looking at a problem; it is not the answer to the problem itself.

## What Makes a Good Framework

A framework is **not**:

- A tip or trick
- A motivational phrase
- A restatement of common sense

A framework **is**:

- Named and attributable ("Schelling Point" beats "focal points")
- Domain-bounded ("Applies in coordination games with multiple equilibria")
- Useful — it changes a decision when applied
- Composable — it stacks with other frameworks

> [!tip] **The lens test.** A real framework, applied to a decision, changes the answer. If it doesn't, it's decoration.

---

## When to Open

- When analyzing a complex problem and looking for the right lens
- During a strategic decision where multiple frameworks apply
- When a familiar problem recurs — check whether a known framework applies
- When teaching or explaining a thought process to AI

---

## How to Use

1. **Browse by situation, not by framework name.** Use the [[wiki/concepts/frameworks/frameworks-moc\|Frameworks MOC]] for situation-based retrieval.
2. **Apply at most two frameworks per decision.** More usually means confused thinking.
3. **Link to outcomes.** Every applied framework should link to a decision in [[06 Decision Archive\|06 Decision Archive]].
4. **Capture new frameworks from [[04 Knowledge Library\|04 Knowledge Library]]** when a domain note reveals a pattern.
