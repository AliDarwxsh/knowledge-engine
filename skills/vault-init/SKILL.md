# vault-init — Knowledge Engine Onboarding

> Run this skill exactly once after cloning knowledge-engine. It transforms an empty vault into a living knowledge system in under ten minutes.

---

## When to Invoke

The user says any of:
- "Set up my knowledge engine"
- "Initialize the vault"
- "Onboard me"
- "/vault-init"
- "I'm new here, where do I start?"

If the user already has a `CLAUDE.md` at vault root, refuse and redirect to [[vault-sync]] instead.

---

## Prerequisites

Before running, verify:

1. **Vault directory exists** with the dual-layer structure (`00 Inbox/` through `13 Meta Analysis/`, plus `wiki/`, `sources/`, `schema/`).
2. **Hermes is installed** and `hermes` command resolves in `$PATH`.
3. **User is in the vault root** (pwd should show `Knowledge Archive Home.md` or equivalent).

If any prerequisite fails, halt with a clear message. Do not proceed.

---

## Execution — Six Phases

### Phase 1: Detect Vault Topology (automated, ~10 seconds)

Scan the vault structure and confirm:

```bash
# Verify all 14 numbered folders exist
for i in 00 01 02 03 04 05 06 07 08 09 10 11 12 13; do
  test -d "${i} *" || echo "MISSING: ${i}*"
done

# Verify machine layer exists
test -d wiki/ || echo "MISSING: wiki/"
test -d sources/ || echo "MISSING: sources/"
test -d schema/ || echo "MISSING: schema/"
```

If any folder is missing, halt and instruct: "Run `bash scripts/setup.sh` first to scaffold the vault."

### Phase 2: Read Governance (automated, ~5 seconds)

Read and internalize:
- `schema/CLAUDE.md` — the three-layer rule architecture
- `schema/ontology.md` — the tag taxonomy
- `schema/agents.md` — agent roles and permission boundaries
- `schema/copywriting-standards.md` — voice standards

These files define how you must write. Internalize them before generating any new content.

### Phase 3: Interview the User (interactive, ~5 minutes)

Ask ONE question at a time. Wait for the answer before asking the next. Do not batch.

Use this exact sequence:

**Q1 — Identity**
"Welcome to your knowledge engine. I'll set this up in under ten minutes.
First: Who are you, and what do you do? Two or three sentences is plenty."

**Q2 — Primary Goal**
"What's the one thing you're building toward right now?
Not a list — the single outcome that would make this year a win."

**Q3 — Communication Style**
"How should I talk to you? Pick the closest match:
- **Direct** — tell me what's true, even when it's uncomfortable
- **Warm** — lead with encouragement, then the hard part
- **Analytical** — show me the data and reasoning
- **Minimal** — just the action, no explanation"

**Q4 — Decision Style**
"When you face a hard decision, what do you usually do?
- Write it out (you think by journaling)
- Talk it through (you think by discussing)
- Sleep on it (you think by letting the subconscious work)
- Build a prototype (you think by doing)"

**Q5 — Active Projects**
"List 1-3 projects you're actively working on. Project name and one-line purpose each."

**Q6 — Capture Volume**
"How often do you have a thought worth capturing?
- Multiple times a day (I'm a heavy thinker)
- Once or twice a day (selective)
- A few times a week (sporadic bursts)"

**Q7 — Existing Knowledge Sources**
"Do you have any of these you want to import?
- ChatGPT conversation export (JSON)
- Claude conversation export
- Other AI chat logs
- Notes from another app (Notion, Apple Notes, Evernote)
- Book highlights / read-later articles

Say 'none' if you're starting clean."

**Q8 — Permission Level**
"How much autonomy should the engine have?
- **Conservative** — classify and link, but never file without my approval
- **Moderate** — classify, link, and file. Flag anything uncertain.
- **Aggressive** — full automation. I trust the engine to organize itself."

### Phase 4: Generate CLAUDE.md (automated, ~30 seconds)

Using the interview answers, write `CLAUDE.md` at vault root. Use this template:

```markdown
---
title: "User Profile"
summary: "Generated by vault-init. Loaded into every Hermes session for persistent context."
date: <YYYY-MM-DD>
tags: [meta, profile, identity]
---

# User Profile

> This file is loaded automatically into every Hermes session. Update it when your situation changes — run `/vault-update-profile` to regenerate.

---

## Identity

<from Q1>

## Primary Goal

<from Q2>

## Communication Style

<from Q3>

## Decision Style

<from Q4>

## Active Projects

<from Q5, formatted as list>

## Capture Pattern

<from Q6>

## Existing Sources

<from Q7>

## Permission Level

<from Q8>

## Engine Configuration

- **Vault path**: <absolute path>
- **Engine version**: 1.0.0
- **Initialized**: <YYYY-MM-DD>
- **Last sync**: <YYYY-MM-DD>

---

## Voice Anchors

- Use [[communication-style]] as the default register.
- When uncertain, ask before assuming.
- Surface contradictions rather than smoothing them.
- Never soften truth for comfort.
```

Write frontmatter, body, and voice anchors. No placeholders.

### Phase 5: Install Cron Jobs (automated, ~15 seconds)

For each of the 5 cron job definitions in `cron/`, install via Hermes:

```bash
# Each cron file is JSON; read and install
hermes cron install < cron/morning-brief.json
hermes cron install < cron/inbox-processor.json
hermes cron install < cron/connection-finder.json
hermes cron install < cron/nightly-lint.json
hermes cron install < cron/weekly-digest.json
```

Verify with: `hermes cron list`

### Phase 6: First Sync (automated, ~2 minutes)

Run the full vault-sync workflow:
1. Scan `00 Inbox/` for any pre-existing items
2. Run vault-lint to establish baseline health score
3. Generate initial graph index
4. Confirm: "Your knowledge engine is live."

## Failure Modes & Recovery

| Failure | Response |
|---------|----------|
| User has existing CLAUDE.md | Halt. Ask: "Replace existing profile, or run /vault-sync to add new sources to current vault?" |
| Vault structure incomplete | Halt. Direct to `bash scripts/setup.sh`. |
| Hermes not installed | Halt. Direct to `https://hermes-agent.nousresearch.com/install.sh`. |
| Cron install fails | Warn but continue. Engine works without cron (manual triggers). |
| User refuses to answer questions | Use placeholders. Mark profile as "minimal" and tell user how to update. |

---

## What This Skill Does NOT Do

- Does not write to `schema/` files (governance is human-owned)
- Does not edit `sources/` files (immutable history)
- Does not delete or rename existing notes
- Does not change the user's permission level after initialization (that requires `/vault-update-profile`)
